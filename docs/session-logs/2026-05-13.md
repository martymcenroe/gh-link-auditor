## Lessons captured

| Date | Lesson | Rule |
|:-----|:-------|:-----|
| 2026-05-13 | A test that "passes locally" but spins on CI is often an `input()` mock returning the wrong value. The Python loop is tight enough to be near-invisible locally; CI's per-test timeout exposes it. | When a test hangs only on CI, check every `patch("builtins.input")` against the actual loop's accepted inputs. `pytest-timeout=60` makes this catchable in ~minutes instead of hours. |
| 2026-05-13 | Trust-state time-bomb tests with hardcoded `merged_at` will rot exactly N days after the test was written (N = the smallest upgrade-window threshold). Same shape as the `test_stargazer_harvester` bomb from the prior session. | All date-driven trust tests use `(datetime.now(timezone.utc) - timedelta(days=K))` where K is well under the smallest window the assertion depends on. |
| 2026-05-13 | `pytest-timeout`'s `--timeout=N` is a hard requirement for any CI job that runs > a few minutes' worth of unit tests. Without it, one bad test eats the whole job budget and the diagnostic value of the run is zero. | Always set a per-test timeout in CI pytest jobs. Set a job-level `timeout-minutes:` too, as a second wall. |
| 2026-05-13 | Worktree-per-issue + classic-PAT script publishing means `git push` from the worktree CAN be rejected post-classic-PAT-write. The remote moves; the local worktree didn't see it. | After running the classic-PAT script, always `git fetch && git rebase origin/<branch>` in the worktree before the next push. Move untracked files aside first if they collide with files landed by the script. |
